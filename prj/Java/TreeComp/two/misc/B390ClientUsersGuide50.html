Chapter 1. Introduction

Build/390 automates the building and packaging of S/390 software products stored in the CMVC library system. Build/390 uses existing S/390 build and packaging tools with minimal changes.

Build/390 consists of a remote client, an MVS server, an RMI server, and a set of tools and processes invoked by the server to process product parts. Build/390 provides a Java client, which runs on Windows platforms, and a UNIX System Services (USS) command line client. The server is an MVS started task that communicates with the client via TCP/IP sockets and can initiate up to 100 simultaneous sub-server jobs to process client requests.

Build/390 provides the following benefits over traditional methods of building and servicing products:

  • A platform-independent Java client
  • An automated process which eliminates numerous manual and tedious steps
  • The ability to restart a build at any phase in the process
  • Error information from the workstation client which eliminates the need to log on to OS/390
  • The ability to save detailed information about builds which enables you to recreate builds quickly and easily
  • The orders and profiles used by Build/390 can be versioned in the CMVC library with the product code.

The client accepts input from the Java client graphical user interface (GUI) or from the USS client command line. You can start a build using the Java GUI dialogs or by entering build instructions on the USS command line. Once all required input information is verified, the client starts a series of build phases that are defined in the BLDORDER order stored in the source library. The client tracks each phase and if any jobs are submitted on the server, it waits until they have all been completed. If all jobs complete successfully, the client starts the next phase. This process continues until all of the predefined phases have been completed. The phases do things such as process macros, compile parts, and link edit.

The build process is completely configurable and can consist of an arbitrary number of phases. For example, a two phase process could consist first of processing all macros by placing them in macro libraries and second of processing all other parts, including all of the modules to be compiled. Any given driver can use any of a number of different build types that specify different phase processing. For example, a final build type may include packaging phases, whereas a development build type would not. The build types and phases are defined in a BLDORDER that is versioned in the library along with other types of orders and source parts.

Chapter 2. Requirements and recommendations

Connection requirements

Ensure that you have TCP/IP installed and a connection to the IBM network.

Minimum hardware requirements for Windows users

  • 128MB RAM
  • 256MB Virtual Memory (paging space)
  • 333 MHz Pentium III or equivalent processor

Build/390 client options

You can run Build/390 commands using the Build/390 client graphical user interface (GUI) or by executing commands from a UNIX System Services (USS) command line.

Build/390 client GUI

The Build/390 client GUI for Version 5.0 runs on Windows platforms only. Windows 95, 98, NT, 2000, and XP are supported. Windows NT, 2000, and XP are recommended.

If you use Windows 95, you may experience problems when running a large build (tens of thousands of parts). Windows 95 does not manage memory well which can result in crashes during a large build. It is important to reboot your Windows system at regular intervals, especially before performing large builds.

USS command line client

Build/390 provides command line support for the USS environment.

BPXPRM00 settings

Parmlib member BPXPRM00 contains customization values for USS. The following settings can affect the USS client and may need to be adjusted by your system administrator:

  • MAXASSIZE: Limits the amount of storage available to the client. A setting of 500M is usually adequate.
  • MAXCPUTIME: Sets the process time limit. It is recommended that this be set to unlimited which is MAXCPUTIME(2147483647).

Build/390 client running in INPUT mode

When running the USS client, you may experience a situation where the client keeps going into INPUT mode. If this occurs, you must periodically press Enter to keep the client running. Your local system administrator should be able to fix this problem.

For instructions, refer your system administrator to Appendix D, "Staying in Running Mode" in the document, Porting Applications to the OpenEdition MVS Platform at http://www.redbooks.ibm.com/redbooks/GG244473.html.

Optional part rename pre-exit

If you use the default LODORDER shipped by Build/390, the Part Rename pre-exit protects you from altering the Build/390 part name. The Build/390 part name is the portion of the file path after the last forward slash and before the first dot.

Note:
If you change the default load order shipped with Build/390 and use a different method to determine the Build/390 part name, the Part Rename pre-exit will not work properly. For example, the MVS server may (depending on the LODORDER) recognize the renamed part as a new part.

Example: The default LODORDER shipped by Build/390 uses several parameters as input. One parameter is the CMVC part name, for example, dir1/dir2/jaker.asm. When the Build/390 LODORDER parses the CMVC part name, it looks for the last forward slash character, then parses the remaining character string into two fields:

PARTNAME
contains the character string after the last forward slash and to the left of the dot. In this example, PARTNAME is jaker.
REST
contains everything to the right of the dot. In this example, REST is asm.

In this example, the Part Rename pre-exit restricts you from renaming the PARTNAME jaker. You can rename those parts of the file path before the last forward slash (dir1/dir2) and after the dot (asm).

For installation instructions, see the Build/390 RMI Server Administrator Guide at http://w3.sdwb.ibm.com/b390doc50/documents50.htm. For information on the default Build/390 LODORDER, see the Build/390 Process Reference at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

Service process considerations

CMVC and BPS

This book describes the Build/390 client's role in the service process. To use the Build/390 Service process, you also need to be familiar with CMVC and BPS. For CMVC information, refer to the CMVC InfoCenter at http://w3.sdwb.ibm.com/cmvc/5.0/infocenter/. For BPS information, refer to the BPS documentation at http://w3.sdwb.ibm.com/bps/bps40Pubs/.

For process flow information and examples, refer to the Build/390 Service Process Guide at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

Concurrent development support

The Build/390 service process supports CMVC Concurrent Development mode.

Note that Build/390 has disabled the CMVC Conflict accept option. If you try to use this option and conflict records exist for the Track, Build/390 will require that you return the Track to the Fix state without saving your changes.

For more information on Build/390 concurrent development support and conflict resolution, refer to Concurrent development service processing.

PassTicket support

The service process requires that your administrator set up PassTicket support on the MVS host system. For more information, refer to the Build/390 RMI Server Administrator Guide at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

APAR part restrictions

By using an optional exit program, Build/390 limits the contents of a CMVC APAR defect on a RETAIN release (FMID) boundary. An optional Build/390 setting allows you to limit the contents of an APAR defect to the RETAIN component of the APAR.

For more information on the optional RESTRICTCHECKINBYDEFECTCOMPONENT setting, see the Build/390 RMI Server Administrator Guide at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

PTF build and force supersede processing

If you are using the force supersede option when building a PTF, see the notes in the ++PTF process section of Running a ++PTF build.

PTF testing

During the PTF test phase, a service builder may want to send a PTF to Test for verification and install testing. For instructions, see Retrieving data from a driver.

The service builder can use CMVC environment lists to customize notification of PTF test readiness. For example, it is possible to configure CMVC with a unique tester for each environment associated with a particular release. When the PTF build completes, the PTF level is set to CERTIFY, the test records are set to READY, and CMVC sends an email notification to each tester of record for all associated environment lists.

Note:
Build/390 will automatically add an environment called B390Internal. Do not take any action on this internal test record. Build/390 automatically sets this record to abstain when all other test records are processed (accepted or abstained.) For an example, see the Build/390 service process example at http://w3.sdwb.ibm.com/b390doc50/B390ServiceProcessExample50.html.

An Environment List is a CMVC option used to specify environments in which a release should be tested and the users who have access to those environments. It includes a list of environment/user ID pairs that indicate users who can test the environment. A test record is issued to each user, and all user test records must be accepted before the PTF ships. For more information on Environment Lists, see the CMVC InfoCenter at http://w3.sdwb.ibm.com/cmvc/5.0/infocenter/.

Chapter 3. Setup and customization

Before you begin

The Build/390 client needs setup information that defines how to communicate with the Build/390 server and the CMVC library server. For the service process, you must set up a connection to the BPS server.

In addition, Build/390 needs information that enables it to run jobs on OS/390. The build processes use the setup information that was in effect when the build page was opened. If you open a new driver build and then change the setup information, those changes will not take effect until you close the driver build and open a new one.

Before using the ZIP function in the Build/390 client, you must authenticate with the RTPGSA firewall. Refer to the following URL for complete information: http://pokgsa.ibm.com/projects/g/gsa_architecture/public/firewall/RTPGSA%20Access%20Steps.html

Starting the client GUI

Setting up server connections

Note:
Be sure to select Save before exiting the Setup dialog or your changes are not saved.

Build/390 server

To set up your Build/390 server connection, do the following:

  1. Select Build/390 > Add.
  2. In the Build/390 Server Setup window, specify the following information:
    Hostname
    The address of the system running the MVS resident Build/390 server, for example, snjeds3.sanjose.ibm.com
    Port
    The port that the Build/390 server is running on. There may be multiple servers running on your MVS system at different levels supporting different level clients, so ask your administrator for the port number that supports your client level.
    TSO user ID
    The ID that jobs will run under. You are prompted for the password for the ID specified here whenever you run the client.
    Account Info
    You can get TSO account information from your TSO logon screen. Check with your Build/390 administrator for the correct format to use.

    For example, on SNJEDS3 in San Jose, you enclose the data in parenthesis and delimit it with commas: 

    (BOB,X12,050,W97)
  3. Click OK to return to the Setup dialog.
  4. Test your connection by Selecting Build/390 > Test Connection from the menu.
  5. Be sure to select Save on the main setup dialog after making your changes.

CMVC library server

To set up your CMVC library server connection:

  1. Select Library > Add.
  2. In the Library Server Setup window, specify the following information:
    Family name
    CMVC family to be used, for example, buildog
    Family address
    CMVC family address, for example, searspoint.sanjose.ibm.com
    Family Port
    CMVC port number, for example, 1242
    User name
    Your CMVC user name, for example, b390id
    Config RMI Port
    The port number of the Build/390 RMI server, for example, 2098. Ask your Build/390 administrator for this port number.
    Use password authentication
    Check this option if you will enter a password when connecting to the CMVC server. If not checked, authorization by the CMVC server is required.
  3. Click OK to return to the Setup dialog.
  4. Test your connection by selecting Library > Test Connection.
  5. Be sure to select Save on the main setup dialog after making your changes.

BPS server

To set up your BPS server connection:

  1. Select BPS > Add.
  2. In the BPS Server Setup window, specify the following information:
    Hostname
    The fully qualified hostname of the BPS service machine, for example, bps17.raleigh.ibm.com
    BPS Port
    The communications port for the BPS service machine

    Tip: If you need information on BPS server settings, your BPS system administrator can generate a report by executing startBPS -info on the BPS-S host system. The BPS PortNo value is the value you enter in this field.

    Release in Service
    A CMVC release, for example, B390TST. Build/390 uses this value and the #SETPDT order to determine the correct PDT (PTF Definition Tool) database for service. For more information on the #SETPDT order, see the Build/390 Process Reference at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.
  3. Click OK to return to the Setup dialog.
  4. Test your connection by selecting BPS > Test Connection.
  5. Be sure to select Save on the main setup dialog after making your changes.

  1. To use the built-in Java viewer, make no selection.
  2. To use a custom editor (recommended), select the Use Custom Editor box, click Browse, and specify the path to your favorite editor.
  3. Be sure to select Save on the main setup dialog when you are done making changes.

Driver build

Overview

A driver build allows you to build a driver using one or more parts that are stored in CMVC. Driver builds use library queries based on user input to determine the list of parts to be built. Parts that are not already up-to-date on OS/390 are then uploaded to the shadow release on OS/390. Metadata that has been set for parts or tracks in CMVC will be used based on your configuration settings. For more information, see Updating configuration data.

The build instructions selected by the user are then run. Prior to running a driver build, the release and driver to be built must have been created on the OS/390 system.

Build phases

The process of building a driver includes all the steps needed to produce a desired package, including part selection, part extract, part upload, compiles, link edits, etc. For a description of the build process and how to specify what each phase does, see the Build/390 Process Reference at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

The client controls driver builds by communicating with the development library and the build server. Before starting a build, you must specify option settings and criteria that you want to use to select the list of parts to build. A Driver build executes the following phases:

Driver Report
Determine if driver is available for build.
Partlist
Determine list of parts to be built via library queries.

As of release 2.4, the partlist generation is done sending a request to the Build/390 RMI server on the CMVC system. All queries are performed from that machine and the resulting list is returned to the Build/390 client. This method improves performance significantly for large products.

Load Order
Determine if the Lodorder has been changed and if so, extract it, upload it and load it into the shadow release.
Check Parts
Check the list of parts against the shadow release and determine which parts need to be updated on OS/390. Also build the list of parts that need to be checked against the driver for information on which parts need to be run through the build phases.
Extract Parts
If the 'Extract parts from library to path' option was specified, extract parts (that need to be updated on OS/390) from the library and ftp them to OS/390.

For the parts to be FTPed to OS/390 in the correct format, they must have been loaded into CMVC properly. Ascii parts must have been loaded as Ascii and the proper codepage must have been used and defined to Build/390. See Code Page Support for more information. Binary parts must have been loaded as and be marked as binary. The LODORDER that you use must properly define how all parts are to be loaded into the shadow. One exception to this is that parts marked as binary in CMVC will be loaded as binary regardless of the LODORDER definition.

Code Page Support
As part of the query process to build the part list, the Build/390 client will obtain the contents of the "Codepage" field in CMVC for every part and pass that value to the Build/390 server as part of the 'Check Parts' process. The server will then consult a site specific table (#CODPAG) to determine the appropriate FTP site command to be used when transferring this file. This command will be passed back to the client as part of the shadow check report. When the client ftp's the part to OS/390, it will ensure that the site command specified is used for that part. Note that there is no way to make use of this value if the part has been shadowed into a DFS hierarchical file system. The USS DFS client uses the same translation table for all parts and it cannot easily be changed.
Load Parts
Load the parts into the shadow.
Driver Check
Determine which parts need to be run through the build phases. This phase can be skipped by selecting the 'Skip Driver Check Phase' check box on the options dialog. In which case all parts would be run through the process.
Host Phases
Run each of the host phases defined in the buildtype selected within the BLDORDER. For details on the host phases that are run for a particular build type, select Option > Build Type Details from the menu on the build dialog.

Handling BLDTYPE that calls XUSERMOD with NOTHINUM=<YES/NO> option

  • BLDORDER has BLDTYPE=(eventually chooses a BLDTYPE that calls XUSERMOD) NOTHINUM=YES

    User attempts to do a driver build (File/New DriverBuild) on a newly created (or existing) BASE driver. No failure; the build proceeds to the next step.

  • BLDORDER has BLDTYPE=(eventually chooses a BLDTYPE that calls XUSERMOD) NOTHINUM=YES

    User attempts to do a driver build (File/New DriverBuild) on a newly created (or existing) thin delta driver based on a FULL Driver (Base or FULL Delta). No failure; the build proceeds to the next step.

  • BLDORDER has BLDTYPE=(eventually chooses a BLDTYPE that calls XUSERMOD) NOTHINUM=YES

    User attempts to do a driver build (File/New DriverBuild) on a newly created (or existing) thin delta driver based on a thin delta driver. The client runs Phase 0, which processes the ORDERs. The client then runs Phase 1 which was defined in the product's BLDORDER. A failure occurs at this point. '*ERROR* ... '

  • BLDORDER has BLDTYPE=(eventually chooses a BLDTYPE that does NOT call XUSERMOD) NOTHINUM=YES

    In all other cases, no errors are produced and builds proceed to the next step.

  • NOTHINUM=NO

    This is the default behavior. In all cases, no errors are produced and builds proceed to the next step.

  • If a user updates the BLDORDER for the driver being built, the updates to the BLDORDER will be used. For example, if you are building a driver based on one where the BLDORDER sets NOTHINUM=YES, but in your new driver, the BLDORDER sets NOTHINUM=NO, then the value of NOTHINUM=NO is used. This is different than when the user chooses the client ++USERMOD. In that case, the changes in the updated BLDORDER would not be used.

For information on what happens when a part fails a build phase refer to Troubleshooting driver builds.

User build

A user build allows you to build a driver using one or more parts that reside on your local workstation or in a PDS on OS/390. The library is not queried for part names or levels, as is done for a driver build. Local parts are uploaded to OS/390 (if not previously uploaded), then the user-selected build instructions are run.

Before you begin

Before running a user build:

  • The release and driver to be built must exist on the OS/390 system.
  • If you are using the Fast Track option, all parts in the user build must have previously been built into the driver or a base of the specified driver. However, if a Model part that was already built is specified, the Fast Track part does not have to be previously built.

The FastTrack option instructs the client to perform a quick compile of the parts selected, then terminate. This option bypasses a number of phases and results in no updates to the driver. For a FastTrack build, the driver must have previously been built. This is so that the ftp characteristics of parts to be uploaded can be determined, information about any models specified can be found, and output libraries can be allocated, such as driver maclibs.

Typically, you would select a Delta Driver as the target for a user build since the primary purpose for user builds is quick turnaround for compiles of updated parts on the workstation, and the user does not want any interactions with other parts in the product. For more information on Delta Drivers, see the Build/390 Process Reference at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

USERMOD builds

You build a USERMOD to update a driver as part of the development process. The MVS Administrator must manage the access to development base drivers intended for USERMOD operations. For more information, refer to the appendix for system administrators in the Build/390 Process Reference at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

USERMODs can also be built using the ++APAR Build process. For more information, see Running a ++APAR build.

USERMODs can be built from one of the following sources:

  • A level containing one or more tracks. The level is processed in the manner described below.
  • A single track build. As long as the track has no unsatisfied requirements, it will be built into a USERMOD using the same steps (except partitioning) as a full level build.

There are two ways to allow a USERMOD with no shippable parts:

  • For an individual product, add EMPTYUM=YES=G to the GLOBAL section of the product BLDORDER.
  • For a site default, set EMPTYUM=YES=G in #ILOCAL2.

For more information about allowing a USERMOD with no shippable parts, refer to the Build/390 Process Reference at http://w3.sdwb.ibm.com/b390doc50/documents50.htm.

The USERMOD process is configurable. Your Build/390 administrator will set up the CONFIG data on a release basis. See Updating configuration data for more information.

Once you select tracks to build into USERMODs, the dependencies between these tracks (prerequisite and co-requisite relationships) are analyzed and the tracks are broken into groups as small as possible, unless the build single USERMOD option is selected, in which case all tracks are built into a single USERMOD, as long as their requirements are met. In this mode, an error in any track will prevent the entire USERMOD from being built.

Beginning with Version 5.0, when the track or level contains parts which are shipped in multiple FMIDs, a separate USERMOD is built for each FMID. However, if you specify HOLD data or IF-REQs, the same HOLD data and IF-REQs are added to each USERMOD. You cannot apply this data to a specific FMID.

Library dependencies fall into three categories:

  • Implicit prereqs

    These are automatically created by the library when a file is updated. For example, file A, version 1 is in track 1. If you later update file A, and put it in track 2, as version 2, then track 1 will be an implicit prereq of it.

  • Explicit prereqs

    This is where a developer has told the library a given track is required before another one can be built. An example of this case is if someone changed file B and checked it into track 3. Someone else was working on file C and checked it into track 4. For C to work correctly, the version of B in track 3 was required. The developer working on file C in track 4 would make track 3 an explicit prereq of track 4 to ensure the right version of B would be available when his version of C was being built.

  • Coreqs

    These can be set explicitly or implicitly.

For more information about library prerequisites and corequisites, refer to the CMVC InfoCenter at http://w3.sdwb.ibm.com/cmvc/5.0/infocenter/.

Using these dependencies, the tracks are broken into groups as small as possible. This allows as many USERMODs as possible to be built, despite failures in the build stage. If all were built into one monolithic USERMOD, an error in a single module would prevent the USERMOD from being built. This way, errors in a track only affect that track and others that depend on it. Other tracks with no dependencies on the bad track can still be built.

Notes:
  1. The build type chosen for the USERMOD process should NOT package a USERMOD using smodbld, because this is done by the client. The phases in the build type are used for running the delta build to prepare the driver for packaging, but NOT to actually package them.
  2. The names of the tracks being built must conform to SMP/E standards. SMP/E requires that any USERMOD name including any co-req or pre-req names be 7 characters long, the first being an alphabetic character and the remainder either alphabetic or numeric. All alphabetic characters must be upper case.
SUPs
Build/390 automatically adds a SUP for each TRACK in the USERMOD. It does not SUP previous USERMODs even if they contain the same set of files. There is no facility to manually add SUP entries.
PREs
To ensure USERMODs are installed in the correct order, Build/390 automatically adds a PRE for the last USERMOD built for each file in this USERMOD. For example, a USERMOD is being built for a track which contains files A and B. Previously, file A was shipped in USERMODs UA00001 and UA00003, and file B was shipped in USERMODs UA00004 and UA00008. So, this usermod will have a PRE for usermods UA00003 and UA00008, the last USERMOD for each file. Note: These prereqs are in addition to any explicit library prereqs specified by the user.

Once the process completes, there will be a new set of drivers on OS/390 and levels in the library holding the USERMODs that have just been built. The levels holding the USERMODs are in commit state.

Service builds

Overview

Build/390 provides facilities to produce ++APARs and ++PTFs. This process involves the Build/390 client and server, CMVC, CMVC user exits, and BPS. BPS polls Retain and opens defects in CMVC when an APAR is created. Using Build/390, CMVC, user exits and BPS, an APAR/PTF can be taken through the service process with RETAIN automatically updated as the APAR/PTF progresses.

This section describes the Build/390 client's role in the service process. To use these processes you should be familiar with CMVC and BPS as well as Build/390.

Error handling in library exit-driven operations

The configuration option, MVSOUTPUTINERRORMAIL, can be used to indicate whether error message(s) for user exit failures will be included in the email message which notifies people that the build has failed.

By default, no error messages are included in the email message (false). Set MVSOUTPUTINERRORMAIL to true to include error messages(s).

The type of failure determines who is notified:

  • For a track-related failure (fix, review, integrate), the defect owner for the defect the track was created against is notified.
  • For a level-related failure (going to commit or complete, post PTF actions), the level owner is notified.
  • For a test record failure, the owner of the test record is notified.

Concurrent development service processing

The Build/390 service process supports CMVC Concurrent Development mode. In concurrent development, a part can be checked out to more than one track at the same time. Concurrent development does not allow more than one user to check out a part to the same track at the same time.

Conflict accept option

Build/390 has disabled the Conflict accept option. If you try to use this option and conflict records exist for the Track, Build/390 will require that you return the Track to the Fix state without saving your changes. If this occurs, Build/390 issues the following messages: 

   In service releases, Build/390 disables the conflict accept option.
   To remove the conflict, return track <track_name> to the Fix state
   and undo any changes to part <part_file_path>. For more information
   on conflict resolution, see the Concurrent Development section
   of the Build/390 Client User Guide.

Conflict resolution

CMVC allows the first user to change a Track for a common part to Integrate. CMVC checks for conflicts when subsequent users attempt to change a Track for the common part to Integrate state. If CMVC detects a conflict, the user is notified and must reconcile the conflict before the Track can move to Integrate state.

Example

In this example, User1 is working on APAR1/Track1/Part2. At the same time, User2 is working on APAR2/Track2/Part2. User1 finishes first and promotes Track1 to Review state. After this occurs, User2 attempts to promote Track2 to Review state. During this process, CMVC detects a conflict in Part2 and rejects User2's Track review. User1 and User2 must resolve the conflict before CMVC will move Track2 to Review state.

For more information on how CMVC handles conflict resolution, refer to the CMVC InfoCenter at http://w3.sdwb.ibm.com/cmvc/5.0/infocenter/.

Chapter 8. Managing metadata

Part metadata

The Build/390 client allows you to manage metadata at the part level. Part metadata can be set for a single part and is saved in CMVC tables. This data is associated with a single part and is versioned just as parts are. Metadata is carried over from one version of a part to the next. If the metadata for a part is changed, that part will be rebuilt the next time it is included in a build.

To set Part metadata, you must have FileModify authority for the part.

When a driver build is started, the Build/390 client will query CMVC for CONFIG information that determines if the part metadata is to be used in the build process. If the processes are turned on in the CONFIG information, the client performs queries to obtain any metadata set for each of the parts to be built. The client will then include these settings (for each part to be built) when building the verb that is used to update the metadata in the UNIMODC.

Build/390 enforces a hierarchy of metadata for the various types and orders. Metadata is defined in the following order:

  1. OP=UPD statements
  2. Local section of the BLDORDER
  3. Part profile
  4. Related metadata part
  5. CMVC part metadata (via Metadata editor)
  6. Embedded part metadata

If any of the levels past the Profile level define a new profile, then that new profile is executed at the time it is defined.

Embedded Metadata is the last set to be applied and cannot be overridden.

To enable the use of part metadata, your Build/390 client CONFIG information must be set properly. This data is set by your Build/390 administrator on a release basis. Contact your Build/390 administrator for more information. The metadata edit menu items are disabled unless the CONFIG data includes the correct settings.

Metadata updates in a concurrent development release

When using part metadata in a release that has concurrent development turned on, all virtual versions of a part in the release (versions that start with v) will have the same copy of metadata.

For example, User1 has Part1 in Track1, and User2 has Part1 in Track2. As long as changes in Track1 and Track2 are virtual, both User1 and User2 will see changes to the metadata when either makes a change. Once the part is given a real version number, it is no longer updated when User1 or User2 updates the metadata for it.

Notification of metadata updates

When metadata for a part is changed, the user who last changed the part is automatically notified by e-mail.

A filtering option enables you to work with only certain parts that meet the criteria you specify. You can specify only one criterion to meet, or multiple criteria. There are three options in filtering parts:

  • Filter based on metadata on the host only. This option filters the matching metadata of parts that exist on the host.

The full set of regular expressions are supported. (Refer to http://www.regular-expressions.info for more information.)

  • Group name (required)

    This serves to partition the different jobs, by username or whatever other criterion one wishes to use.

  • Event name (required)

    A name for this individual event. Should be descriptive.

  • Interval (required)

    00:00 means just run once per days selected, otherwise, count as a delay between consecutive executions of the job. 01:00 means every hour, 00:40 means every 40 minutes, etc.

  • Start (required)

    Determines the time of the first execution during the day. 00:00 is midnight, 09:13 is 9:13, etc.

  • Stop

    Determines the time to stop scheduling an event. Say start = 09:00, interval is 00:20, and stop time is 10:10. The job will run at 9, 9:20, 9:40, and 10.

  • Days to run (required)

    Determines the days an event is scheduled.

  • Return code checking

    If selected, after an execution of the job the return code is checked. If it is greater than the value entered here, a notification is sent to the contacts on the contact list.

  • Step RC checking

    If selected, after an execution of the job, the return code of the given step name is checked. If it is greater than the value entered here, a notification is sent to the contacts on the contact list.

  • DSN (required)

    This points to the actual data set name of the JCL file on OS/390 to run at the specified times. The JCL must meet the following requirements:

    • The JCL must NOT contain either the USER= or PASSWORD= keywords.
    • The JCL may contain multiple steps, but no more than one JOB statement.
    • The owner of the server must have RACF read authority to the data set specified for DSN.
  • Contact list (required if RC or Step RC checking selected)

    This is a list of node.userid entries that receive a notification when a job fails.

Remove
Removes the selected job from the scheduler queue.
Edit
Edit works like ADD, but all fields start off filled in to the values from the selected job. Also, group name and event name cannot be changed.
Reactivate
This can only be used on jobs that are scheduled but not currently active. It sets them to the active state.
Deactivate
This can only be used on jobs that are scheduled and are currently active. It sets the state to inactive so the job will no longer be run. Inactive jobs remain in the queue.

Chapter 10. Troubleshooting and Help

Obtaining error information

When an error occurs, the client attempts to display enough information to communicate what happened. For more detail:

  • Refer to the bottom of the Build390.log file in the \misc directory.
  • Each build has its own log file. All information is logged both to the main log file and to the log file for that specific build. To view the log for a build, either select the build in the Load or Cleanup dialog and select View Log, or select View Log from the build dialog.
  • Error dialogs now include a button that will display the build log.
  • Check the Java window for error information.
  • The client also provides options you can use to get more information about an error. For details, refer to Optional parameters.

Reporting errors to development

If you think you have a problem with Build/390,

  1. First analyze the error information that you have.
  2. If you are not able to resolve the problem, contact the Build/390 support person for your location.
  3. Your support person will assist you in trying to resolve the problem.
  4. If the problem appears to be a Build/390 problem, your support person will open a problem in the SDWB database. You will need to assist him by providing as much error information as possible.
    Note:
    You can Zip the build information and transmit it to development for analysis by selecting menu items on the Tools > ManageProcesses > Action > Zip dialog.

Common problems

Build is hung

My build completed part of the phase that extracts and uploads parts but now appears to be hung.

The Build/390 client keeps 5 threads running (extracting parts from the library and uploading them) until all parts have been processed. This condition can occur when the CMVC server stops responding to the extract requests or the FTP server on MVS stops responding.

Terminate the Build/390 client and restart it with the /remote option. When asked if you want to restart the previously run build:

  1. Click Restart.
  2. Restart at the phase, 'Update files in MVS shadow database'.
  3. Click Yes to start extracting and uploading.

The client will start extracting and uploading where it left off, but will now have just 1 thread communicating with the CMVC server and FTP server instead of 5.

Build status message has asterisks appended to it

I am waiting for the Build/390 server to respond to a request and the status message at the bottom of the client window has asterisks appended to it.

This error occurs when the requested function is taking a long time to return. Often this is because a job was submitted as a result of the request and ended up on the input queue waiting to be run. The server is responding with time outs that are indicated by the asterisks. To verify that a job is waiting to be run, logon to the MVS userid specified in setup and check the job queue.

Build/390 client will not accept your password

I am trying to run from the command line on Windows 95 or Windows 98, and the client will not accept my password.

Windows 95 and Windows 98 do not work well with Java when trying to mask a password entry during prompting. To fix this problem, specify the password using the pw_userinfo=<password>pair as part of the command or in the parmfile.

Build/390 command will not complete successfully on Windows 95 or Windows 98

If you are using Windows 95 or Windows 98, be sure to enclose command parameters in double quotes as shown in Syntax for supported platforms.

Errors when Build/390 client queries the library

I am getting errors when the Build/390 client tries to perform library queries, or I am getting empty query results when I know there should be results.

Either of these conditions can occur if you do not have your setup data specified correctly for the library. Make sure that the family name, family address, port, and username are specified correctly. Also make sure that you can successfully run library commands from the command line, for example, Report -view fileview. If you cannot run library commands, check your environment variables for the library.

Message from Build/390 server that a job number is flushed

I get one of the following messages from the Build/390 server: 

CLSVnnnn(JOB12345) FLUSHED. CHECK USERID/PASSWORD
or
ELAPSED INTERVAL OF 00060 SECONDS EXPIRED. "job number"
IS LOST!! PROBABLY FLUSHED SOMEHOW!!!!

This error occurs when a request to check on the status of a job is submitted by the client and the job is not found, or when a job gets submitted as the result of a client request and the job gets flushed. It can occur if:

  • The specified userid, password, or accounting information was not valid.
  • There is an error in the Server setup (see your administrator).
  • There has been a RACF failure.

Message - Cannot open message catalog

I get the following message when the client tries to communicate with CMVC: Cannot open message catalog. Setup NLSPATH.

This error occurs when the NLSPATH environment variable is not set correctly. If you have more than one copy of CMVC installed on your work station, you may need to add a SET statement to your Build390.bat or Build390.ksh file. Add a line that sets the NLSPATH environment variable to the correct path.

Message - Can't find class Build390/MBClient

I get the message, Can't find class Build390/MBClient, when I try to start the client.

Be sure to start the client by running Build390.bat .

If you are starting the client correctly, there may be a problem with your Build/390 installation. Contact Build/390 support for assistance.

Message - Error loading <buildid>.ser file

An error occurs when the client attempts to load a build.

This error can occur after installing an upgrade to the client. It happens because the Java class that defines what a build is, has been altered. The client was trying to load a build from a previous invocation that was not run to completion. At this point there is no work around, other than to recreate the build from the beginning.

Message - Out of environment space

I am running Build/390 with CMVC password support, using the CMVC 1.8 client on Windows 95 with a dial-in dynamic IP address. CMVC rejects the connection request with a message that I am not in the host list.

The following message is displayed in the MS-DOS window: 

Out of environment space

To fix this problem:

  1. Check with your CMVC administrator to ensure you have a valid password for your CMVC ID.
  2. Start the CMVC 1.8 client and verify you can access the Library using this password.
  3. Start the Build/390 client.
  4. Check the MS-DOS window where the Build/390 batch file ran for the error message, Out of environment space, after the SET CMVC_AUTH_METHOD=PW was issued from your batch file.
  5. To increase the environment space for Windows 95, use one of the following methods. The first method will change the setting for Build/390 only; the other methods will change the setting for all MS-DOS programs in your Windows 95 start menu.
    • To change the setting for Build/390 only:
      1. Right click the Build/390 program and select Properties.
      2. Click the Memory tab and look for the box labeled Initial Environment. The default value is Auto. Change the value to 1024.
      3. Click OK to close the Properties window.
    • To change the setting for all MS-DOS programs listed on your Start menu:
      1. Right click the Windows 95 Task Bar and select Properties.
      2. Select the Start Menu Programs tab; then click Advanced.
      3. Right click the MS-DOS shortcut and select Properties.
      4. Click the Memory tab and look for the box labeled Initial Environment. The default value is Auto. Change the value to 1024.
      5. Click OK to close the Properties window.

      Another method is to include a record in your CONF.SYS file similar to the following example: 

      SHELL=C:\COMMAND.COM /E:2048
/P

      In this example, 2,048 bytes is allocated for environment space.

Service build error - APAR does not exist

I successfully built a ++APAR, but an error occurred later in the service process which says that the APAR object does not exist.

This error can occur if the APAR build or setup object on the RMI server is corrupted or deleted. The service process fails. To fix this situation:

  1. Start the Build/390 client.
  2. Select Tools > Manage Processes to find the build that was run from your client.
  3. Select the build that relates to the failed APAR, then File > Copy the RMI object to the RMI server.

Metadata editor error about all lists containing the same number of entries

I am making Metadata changes using the metadata editor and get the following error: All lists in the Related entry group must contain the same number of entries.

This error occurs if you do not have the same number of entries in each of the following fields: DISPLN, DISTLIB, DISTSRC, FCNS, PARTTYPE, SHPALIAS, and SYSLIB.

The multiple entry fields DISPLN, DISTLIB, DISTSRC, FCNS, PARTTYPE, SHPALIAS, and SYSLIB must all have the same number of values. The n'th value in each field defines a shippable for the part. DISTNAME is also part of the group, but a single entry is permitted and in that case it is used for all entries.

Windows 95 crashes when you run a large build

Windows 95 does not manage memory well which can result in Windows 95 crashing during a large build. If you are going to be running large builds on a windows platform we recommend using Windows NT, Windows 2000, or Windows XP.
